Kubernetes port details

In Kubernetes a Service has up to three port fields. Knowing the difference is the key to understanding the table below:

  • port — the port the Service listens on (what other pods call).
  • targetPort — the container port traffic is forwarded to.
  • nodePort — only for NodePort/LoadBalancer; the port opened on every node (auto-assigned 30000–32767 unless set).
Component Service type Service port targetPort (container) DNS name (in-cluster) Reachable from
Frontend LoadBalancer 80 8501 frontend.langgraph-app.svc.cluster.local Internet (via ELB external IP)
Backend ClusterIP 8000 8000 backend.langgraph-app.svc.cluster.local Inside cluster only
Redis ClusterIP 6379 6379 redis.langgraph-app.svc.cluster.local Inside cluster only
Why the frontend Service uses port 80 → 8501: the public LoadBalancer exposes standard HTTP port 80 to users, then forwards to the Streamlit container which actually listens on 8501. Backend and Redis keep the same number on both sides because nothing external talks to them.
Service discovery: pods reach each other by Service name, not IP. Within the langgraph-app namespace, http://backend:8000 and redis://redis:6379/0 resolve automatically via Kubernetes DNS — which is exactly what BACKEND_URL and REDIS_URL are set to.

Scaling: vertical vs horizontal

Two independent axes. Horizontal = more pods; vertical = bigger pods (or bigger nodes).

Horizontal scaling (more replicas)

Add more identical pods behind the same Service. Best for stateless tiers (frontend, backend).

# Manual: change replica count
kubectl scale deployment/backend  --replicas=4 -n langgraph-app
kubectl scale deployment/frontend --replicas=4 -n langgraph-app

# Automatic: Horizontal Pod Autoscaler (uses metrics-server, already installed)
kubectl autoscale deployment backend  --cpu-percent=70 --min=2 --max=6 -n langgraph-app
kubectl autoscale deployment frontend --cpu-percent=70 --min=2 --max=6 -n langgraph-app
kubectl get hpa -n langgraph-app
Do not horizontally scale redis here — it is a single-writer cache backed by one ReadWriteOnce EBS volume. More replicas would each want their own volume and would not share cache state. Use Redis replication / a managed service (ElastiCache) if you need HA Redis.

Vertical scaling (bigger pods / nodes)

Give each pod more CPU/memory, or use larger nodes.

# Per-pod: raise requests/limits in the manifest, then re-apply
#   k8-deployment/03-backend.yaml
resources:
  requests: { cpu: "250m", memory: "512Mi" }
  limits:   { cpu: "1",    memory: "1Gi"  }

kubectl apply -k k8-deployment      # rolls pods with the new sizes

# Per-node (EKS Auto Mode): the cluster auto-provisions right-sized nodes.
# On a managed node group instead, swap the instance type (e.g. t3.large) or
# add nodes so larger pods can be scheduled.

Cluster / node scaling

On EKS Auto Mode (this cluster) node scaling is automatic — when pods cannot fit, Auto Mode launches more/larger nodes and removes them when idle. No Cluster Autoscaler or node group to manage. On a classic cluster you would scale a managed node group manually or with the Cluster Autoscaler / Karpenter.

Load balancing

There are two layers of load balancing in this stack:

LayerWhat balancesHow
External (internet → frontend) User traffic across the 2 frontend pods AWS Network Load Balancer, provisioned by EKS Auto Mode via loadBalancerClass: eks.amazonaws.com/nlb, targeting pod IPs directly (nlb-target-type: ip)
Internal (frontend → backend) Requests across the 2 backend pods Kubernetes ClusterIP Service backend — kube-proxy spreads connections across the backend Endpoints
Internal (backend → redis) Single redis pod ClusterIP Service redis (one endpoint; no balancing needed)
Why an NLB and not a Classic ELB: on EKS Auto Mode a plain type: LoadBalancer falls back to a Classic ELB that registers nodes as targets — which Auto Mode does not wire up, so it times out. The eks.amazonaws.com/nlb class makes Auto Mode's own controller build a public NLB that points straight at the frontend pod IPs. See k8-deployment/04-frontend.yaml.

Where the data is stored

DataStored inPersistence
Redis cache (LLM responses) Redis in-memory, persisted to /data via AOF onto an Amazon EBS volume bound by the redis-data PVC Survives pod restarts (volume re-attaches). 24h TTL per key (CACHE_TTL_SECONDS).
Frontend chat history Streamlit st.session_state — browser session only Ephemeral; lost on refresh / new session
Backend / frontend app code Container image layers (read-only) Immutable; no app-written state on the pod filesystem
Secrets (API key) Kubernetes Secret in etcd (see below) Cluster-lifetime
Only Redis has durable storage (the EBS-backed PVC). Frontend and backend are stateless, which is exactly why they can be horizontally scaled freely while Redis stays a single replica.

How the .env / secrets are processed

The same .env is consumed differently depending on where the app runs. In both cases the file itself is gitignored and never committed.

EnvironmentMechanismHow the key reaches the app
Local / Docker Compose Docker reads .env docker-compose.yml has env_file: .env on the backend service → Docker injects each line as a container environment variable at startup.
Kubernetes (EKS) Kubernetes Secret kubectl create secret generic backend-secrets --from-env-file=.env turns the file into a Secret stored in etcd. The backend Deployment maps it in via secretKeyRef (GROQ_API_KEY) so the pod sees it as an env var. 
DOCKER COMPOSE                          KUBERNETES (EKS)
──────────────                          ────────────────
.env  ──env_file──►  backend            .env ──from-env-file──►  Secret "backend-secrets" (etcd)
container env                                                         │ secretKeyRef
                                                            backend Pod env  GROQ_API_KEY

App code is identical: utils/llm_call.py just reads os.environ["GROQ_API_KEY"].
So: it is Docker locally, Kubernetes in the cluster. The application code does not know the difference — it only reads environment variables. Docker handles injection from .env on your machine; in EKS a Kubernetes Secret does it. Note Kubernetes Secrets are only base64-encoded in etcd by default; for stronger protection enable envelope encryption (KMS) or use the External Secrets Operator / AWS Secrets Manager. The plaintext key only ever lives in the gitignored .env — see k8-deployment/01-backend-secret.example.yaml for the template.
Local vs Kubernetes: with docker compose all three ports are published to localhost (8501 / 8000 / 6379). In Kubernetes only the frontend is public; the backend and Redis stay internal (ClusterIP). To reach an internal service while debugging, use kubectl port-forward svc/backend 8000:8000 -n langgraph-app.
